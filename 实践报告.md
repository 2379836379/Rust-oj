# Rust-oj 实践报告

## 1. 项目背景与目标

Rust-oj 是一个面向 OpenJudge 的桌面客户端项目，采用 **Rust（Tauri）+ TypeScript（WebView UI）** 的技术路线，实现登录、课程/比赛浏览、题目查看、代码测试与提交、提交结果获取与轮询、收藏与缓存管理等功能。

本次实践的核心目标是：
- 在桌面端提供稳定的 OpenJudge 使用体验；
- 形成清晰的模块边界与可维护的工程结构；
- 实现题目页的“测试—提交—结果”闭环，并保证会话与缓存能够长期保持。

## 2. 技术选型与理由

### 2.1 总体架构
- **Tauri**：用 Rust 承担系统能力（网络、持久化、SQLite、文件访问），用 WebView 提供跨平台 UI。
- **TypeScript + Vite**：开发效率高，适合组件化页面与交互；并能保持页面级结构清晰。

### 2.2 网络与会话
- **reqwest**：Rust 侧 HTTP 客户端。
- **reqwest_cookie_store + cookie_store**：支持 CookieJar 与持久化存储，便于实现“长期保持登录”。

### 2.3 HTML 解析
- **regex**：用于从 OpenJudge 页面中提取关键字段（链接、标题、状态等）。
- 解析策略强调：尽量使用结构性锚点、减少对站点文案（中文/英文）的依赖，并避免使用 Rust regex 不支持的 look-around。

### 2.4 本地存储与缓存
- **rusqlite（bundled）**：缓存、收藏与登录信息等结构化数据落 SQLite。
- 配置（OpenAI config / AppState）使用 **TOML**，便于人工阅读与编辑。
- Cookie 持久化保存为 `openjudge_cookies.json`（位于用户配置目录）。

## 3. 工程结构与模块划分

目录结构：
- `r/oj-client/src/`：前端 UI（TypeScript），按“分页结构”组织，单页单文件。
- `r/oj-client/src-tauri/`：Rust 后端，包含 command 注册、网络、解析、缓存、存储等模块。

关键模块职责：
- `src-tauri/src/main.rs`：注册 Tauri commands，对外提供统一 API。
- `src-tauri/src/state/`：`AppCtx` 聚合业务逻辑与会话状态，提供各类 command 的实现。
- `src-tauri/src/network/`：`OpenJudgeSession` 负责 HTTP 请求、默认头、cookie 合并等细节。
- `src-tauri/src/parser/`：各页面 HTML 解析器（Class/Contest/Problem/Submit/Result）。
- `src-tauri/src/cache/`：页面缓存仓库（SQLite）。
- `src-tauri/src/favorite/`：收藏夹仓库（SQLite）。
- `src-tauri/src/storage/`：存储统计、缓存清理、登录缓存等。

前端通过 `invoke()` 调用 Rust commands，保持 UI 层不直接处理 HTML 解析与鉴权细节。

## 4. 关键功能实现过程

### 4.1 Tauri command 设计
项目将能力以 command 的形式提供给前端，典型命令包括：
- 登录/会话：`oj_login`、`oj_logout`、`oj_get_joined_classes`
- 页面打开：`oj_open_class`、`oj_open_contest`、`oj_open_problem`
- 提交闭环：`oj_open_submit`、`oj_submit_solution`、`oj_open_result`、`oj_result_is_waiting`
- 本地测试：`oj_judge_source`
- 收藏/存储：`oj_favorite_*`、`oj_storage_*`

该方式的优势：
- UI 与业务逻辑隔离，便于调试与迭代；
- 网络与缓存可以在 Rust 侧统一处理，减少前端重复逻辑；
- command 输出结构化数据，降低 UI “对页面结构变动的敏感性”。

### 4.2 会话保持与 Cookie 持久化
实践中，OpenJudge 的登录态主要依赖 Cookie。为实现“长期保持登录”，采取了两层策略：
1) **内存 session 优先**：应用运行期间尽量复用同一个 `OpenJudgeSession`（类似一个全局 CookieJar）。
2) **磁盘持久化**：关键请求后将 cookie store 写入 `openjudge_cookies.json`，下次启动可恢复。

此外，为提高稳定性，实现了“自动重登”机制：当响应被识别为登录页时（例如包含 `OpenJudge - Login` 或“你还没有登录”），会尝试使用登录缓存进行一次重登，并重试原请求。

### 4.3 提交闭环：Submit → Result
题目页需要完整支持：
- 读取 submit 表单（contestId / problemNumber / sourceEncode / language / action）。
- 构造提交 payload（source 使用 base64，表单 urlencoded）。
- 调用 OpenJudge `submitv2` 接口提交。
- 获取提交后结果页 URL，并拉取结果页解析出判题状态。
- 当状态为 Waiting 时，按固定间隔轮询刷新。

#### 4.3.1 Submit 页解析与 payload 构造
在 `parser/submit.rs` 中解析：
- `<form id="solution_submit" action="...">` 获取 action URL
- 隐藏字段 `contestId`、`problemNumber`、`sourceEncode`
- language radio 列表 `value/checked/label`

在构造 payload 时：
- 选中语言（默认使用 checked 项）
- source 进行 base64
- 使用 `application/x-www-form-urlencoded` 发送

#### 4.3.2 Result 页解析与轮询
在 `parser/result.rs` 中解析：
- `<div id="pageTitle"><h2>#<id>...` 获取 submission id
- `<p class="compile-status"> ... <a href="..." class="result-...">TEXT</a>` 获取状态、链接、class

当状态为 `Waiting`：前端/后端以 2 秒间隔重拉结果页，直到出现终态。

### 4.4 关键问题与解决：跨子域结果页导致“无状态”

实践中发现：提交后结果页可能跳转到 `*.openjudge.cn` 子域，例如：
- `http://cxsjsx.openjudge.cn/practise2026py/solution/52872585/`

如果请求该子域时没有带上 root 域（`openjudge.cn`）的 cookie，会被视为未登录，从而拿到的页面不包含真实判题信息，导致 UI 显示 `Judge Status: <none>`。

解决策略：在网络层实现 **跨子域 cookie 合并**：
- 当请求 host 为 `*.openjudge.cn` 且不等于 `openjudge.cn` 时，从 cookie store 中读取 `openjudge.cn` 的 request cookies，并 best-effort 合并到当前子域请求所用的 cookie 集合中。

该策略能够保证：提交跳转到任意 `*.openjudge.cn` 子域的结果页时，仍然可以访问到完整的状态页面并完成解析。

### 4.5 本地测试（OJ-judger）
题目页支持本地测试：
- 通过 command `oj_judge_source` 将代码与 stdin 发送至 OJ-judger。
- judger 地址在 Rust 侧以常量形式固定配置。

### 4.6 收藏、缓存与存储管理
- 收藏夹：SQLite 持久化保存题目元信息与保存时间，并支持按 folder 管理。
- 页面缓存：对 Class/Contest/Problem 等页面信息做缓存，在网络失败或刷新阶段提供兜底。
- Storage 页：提供缓存占用统计与清理入口，支持一键清除缓存。

## 5. 开发与运行流程

### 5.1 环境要求
- Node.js 18+
- Rust stable toolchain

```powershell
cd r/oj-client
npm install
npm run tauri dev
```

```powershell
cd r/oj-client/src-tauri
cargo check
```

## 6. 实践成果与效果

- 已完成桌面端的核心闭环：
  - 登录 → 浏览课程/比赛/题目 → 本地测试 → 提交 → 拉取并显示判题结果（含 Waiting 轮询）
- 具备较强的可维护性：
  - Rust 侧统一处理网络、cookie、解析、缓存
  - 前端以 page 为单位组织，交互清晰
- 针对真实站点行为做了关键适配：
  - `*.openjudge.cn` 子域结果页的 cookie 合并
  - session 过期的自动重登与重试

## 7. 经验总结

1) **会话/跨域是桌面客户端的核心坑点**：浏览器环境往往自动处理 domain cookie；桌面端若用自建 cookie store，需要显式处理子域行为。
2) **HTML 解析应尽量“结构化”而非依赖文案**：站点语言/文案会变，但 DOM 结构相对稳定；并且 Rust regex 不支持 look-around，需要设计更朴素的正则。
3) **命令边界清晰可显著降低耦合**：让 UI 只关心数据和渲染，把网络/解析/缓存收敛在 Rust 层。
4) **持久化策略要分层**：运行期使用内存 session 保证一致性，退出后用 cookie 文件/SQLite 保证可恢复。
